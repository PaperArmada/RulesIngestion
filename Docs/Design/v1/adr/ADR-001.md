# ADR-001: EvidenceUnits Are Canonical Admissible Layer

**Status:** Accepted (amended 2026-03)  
**Date:** 2026-02  
**Amendment:** 2026-03 — GoldAnchor layer for benchmark stability

## Context

The pipeline produces multiple representations: raw prose, structural blocks, EvidenceUnits, and retrieval-only projections (e.g. clause families). Downstream systems (Stage C, answers, citations) need a single source of truth for what counts as evidence.

## Decision

EvidenceUnits emitted by Stage B are the **canonical admissible layer**. Every fact or claim that requires a citation must cite one or more EvidenceUnits. No other artifact (projections, enrichment, graph nodes) may be cited as evidence unless it is explicitly tied to an EvidenceUnit and the citation resolves to that unit.

## Consequences

- Retrieval may use projections to improve ranking and recall, but the retrieved items that are presented as evidence must be EvidenceUnits (or clearly mapped to them).
- Stage C enrichment must not alter EvidenceUnits; enrichment is non-authoritative and has its own schema and provenance.
- Gold sets in Retrieval Lab are **resolved** sets of EvidenceUnit IDs, derived from GoldAnchors (see Amendment below).

## Amendment: GoldAnchor Layer (2026-03)

### Problem

EvidenceUnit IDs are derived from text content and structural path (`blake3(text | path)`). Any change to chunking strategy — adjusting `min_chars`, changing merge thresholds, improving heading extraction — changes unit IDs, breaking benchmark gold references. This causes chunking improvements to masquerade as benchmark regressions.

Stage B determinism guarantees reproducibility for a fixed substrate configuration, not benchmark stability across configuration changes.

### Decision

Adopt a two-layer gold model:

1. **GoldAnchor** — stable editorial anchor representing an authored evidence region relevant to a benchmark query. Captures provenance-rich editorial identity (page, structural path, representative quote, cached lineage IDs). Assigned once at authoring time; identity persists across chunking changes.

2. **EvidenceUnit ID** — current admissible, citable runtime unit in a specific substrate build. Remains the only layer retrieval is scored against.

Benchmark gold is authored against GoldAnchors, not directly against current EvidenceUnit IDs. At eval time, GoldAnchors are deterministically resolved to current EvidenceUnit IDs for the target substrate version.

### Vocabulary

| Term | Definition |
|------|-----------|
| **GoldAnchor** | Durable editorial reference to an authored evidence region. Identity is opaque and persistent. Contains page, structural_path, anchor_quote, and cached provenance signals. |
| **AnchorResolution** | The deterministic mapping from a GoldAnchor to one or more current EvidenceUnit IDs in a particular substrate version. |
| **ResolvedGoldSet** | The runtime set of EvidenceUnit IDs produced by anchor resolution, used for retrieval scoring. |
| **AnchorCoverageStatus** | Whether the current substrate resolved the anchor: `exact`, `split`, `merged`, `approximate`, or `unresolved`. |

### Resolution Contract

At eval time:

1. Load benchmark queries and their GoldAnchors.
2. Resolve all anchors against the current substrate (deterministic, logged, auditable).
3. Materialize `required_gold` / `supporting_gold` as current EvidenceUnit IDs.
4. A ratified eval run requires all required anchors to resolve successfully.
5. Score retrieval against the resolved EvidenceUnit IDs.

Unresolved anchor drift becomes a **benchmark resolution failure**, not a false retrieval miss.

### Resolution Ladder

Anchors are resolved in priority order:

1. Cached unit IDs still present in corpus (exact).
2. Source unit ID lineage traversal via `source_unit_ids` (handles fold/merge).
3. Page + structural_path + quote overlap matching (approximate).
4. Nearby page (±1) + structural_path + quote fallback (approximate).
5. Unresolved — no acceptable match.

### Scoring Semantics

When a required anchor resolves to multiple units (split):

- **anchor-hit policy** (default for T1/atomic): any resolved unit satisfies the anchor.
- **full-anchor policy** (compositional queries only): all resolved units must appear.

### Anchor Sharing

GoldAnchors are per-document and may be shared across multiple benchmark queries. The benchmark file carries a top-level `anchors` dict; queries reference anchors by `anchor_id` in `required_anchor_ids` / `supporting_anchor_ids`.

### Compatibility

Existing fields (`required_gold`, `supporting_gold`, `gold_unit_ids`, `gold_locations`) are preserved as derived/cached runtime fields. The GoldAnchor layer is additive.

- Schema: `retrieval_lab/anchor_schema.py`
- Resolver: `retrieval_lab/anchor_resolver.py`
- Enrichment: `scripts/enrich_benchmarks_with_anchors.py`
